**Core design: PENDING_COMPARISON edges (inverted tracking)**

Instead of building COMPARED edges to track "already checked, no relationship" (which pollutes the graph), we use PENDING_COMPARISON edges to track "not yet compared":

- New decision written → `createPendingEdges()` creates PENDING edges to all existing active decisions
- `connectDecisions()` processes PENDING edges → LLM grouping + per-group relationship analysis → creates relationship edges where found, deletes all PENDING edges in batch (regardless of result)
- Decision content updated → `invalidateDecisionEdges()` deletes all relationship + PENDING edges, rebuilds PENDING edges
- Graph converges to clean state: only meaningful edges remain (CAUSED_BY, DEPENDS_ON, CONFLICTS_WITH, CO_DECIDED)

**Exported functions:**
- `createPendingEdges(session, newIds)` — Build PENDING edges after writing decisions
- `invalidateDecisionEdges(session, decisionId)` — Reset a decision's edges after content update
- `connectDecisions({ dbSession, ai, budget, batchCapacity, concurrency })` — Process all PENDING edges
- `getPendingStatus(session)` — Dashboard query for PENDING edge stats
- `normalizeKeywords(session, ai)` — Global keyword normalization (should run before connectDecisions)

**Pipeline integration order:**
```
batchWriteDecisions(...)     // Write decisions
normalizeKeywords(...)       // Normalize keywords (improves grouping accuracy)
createPendingEdges(...)      // Mark new decisions as needing comparison
connectDecisions(...)        // Process PENDING edges (grouping + LLM relationship)
```
